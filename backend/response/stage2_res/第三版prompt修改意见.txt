================================================================================
第三版 Prompt 修改意见：以判决主文为锚的流程挖掘
================================================================================

SYSTEM_PROMPT 
【关键法律术语定义（请严格据此理解，不得自行扩张或限缩）】

  1. 诉讼标的
     指本案中原告请求法院予以裁判的最核心的法律关系或请求权。
     例如：在离婚诉讼中，诉讼标的是「原告请求解除的婚姻关系」。
     注意：一个程序只有一个诉讼标的。不得将支撑该标的事实或证据误认为标的本身。

  2. 判决主文
     指判决书中，法院对诉讼标的作出的最终、有直接法律效力的裁断结论。
     例如：「准予...离婚」、「驳回...全部诉讼请求」。
     注意：判决主文是唯一的、终局性的。你的任务，就是复原通向这一句话的逻辑路径。

  3. 裁判理由
     指判决书中，支撑判决主文成立的事实认定、证据评价和法律适用论证。
     注意：裁判理由中包含了对各种「法律事实」是否存在的认定，这些认定是为判决主文服务的
     「论证步骤」，不是独立的判决结果。

  4. 法律事实
     指法律规定的、能够引起法律关系产生、变更或消灭的客观情况。
     例如：「分居满二年」是引起离婚请求权产生的法律事实。
     注意：法律事实是法院审查的对象，但它本身不是诉讼标的，也不是判决主文。

  5. 本次程序
     指当前你正在分析的这个法律文书所记载的、具有独立案号的单一审判或审查程序。
     注意：你提取的所有步骤，必须全部发生在本次程序之内。任何需要「另行起诉」、「另案处理」、
     「可上诉」、「申请再审」的内容，都属于独立的另一程序，必须排除。
    
约束："步骤描述必须基于原文的具体裁判说理，不得仅转述结构分析中的 summary。"

二、逻辑链条：从判决主文反推裁判理由
================================================================================

这五个术语构成一条严格的单向依赖链：

   法律事实 ──(被审查)──→ 裁判理由 ──(支撑)──→ 判决主文 ──(回应)──→ 诉讼标的

   本次程序 ─────────────────────(容器)──→ 以上全部

流程挖掘的本质就是：从「判决主文」出发，逆推「法院审查了哪些法律事实、形成了怎样的裁判理
由、最终得出这个判决主文」。其他一切内容的唯一判定标准是：

   它是否出现在通向判决主文的裁判理由中？

   是 → 保留为步骤
   否 → 丢弃


三、举例：如何用这套术语消除误判
================================================================================

以 14 种感情破裂法定情形中的第 7 条为例：

  原文：
  「因感情不和分居已满3年，确无和好可能的，或者经人民法院判决不准离婚后又分居满1年，
   互不履行夫妻义务的。」

  错误读取（没有术语约束时 LLM 的直觉）：
    看到「判决不准离婚后」→ 理解为「前一次程序结束后，再启动另一个程序」→ 生成回边和环

  正确读取（用本术语体系约束后）：
    - 诉讼标的：解除婚姻关系
    - 判决主文：准予离婚 / 不准予离婚
    - 本句性质：「分居满1年」是一个法律事实
    - 审查方式：判断「在本次程序启动之前，是否存在『前次判决不准离婚 + 分居满1年』这一
      已成就的法律事实」
    - 功能定位：如果该法律事实成立 → 写入裁判理由 → 推导出判决主文「准予离婚」
    - 结论：这是裁判理由中的一个论证步骤，不是程序跳转。不创建新节点，不形成回边。

  自检通过标准：
    Q: 「分居满1年」是否为本次程序的判决主文服务？
    A: 是 — 它属于感情破裂的法定情形之一，直接支撑判决主文。
    Q: 它是否启动了另一个独立程序？
    A: 否 — 它只是本次程序中法院审查的一项法律事实。
    → 保留为步骤。

当前版本提示词，stage2整体表现良好
