================================================================================
第四版架构修正：从"单模板"到"以诉讼标的为根的子模板树"
================================================================================

一、根本缺陷
================================================================================

当前 Stage3 的隐含假设：
  「一个关键词 → 一个通用流程模板」

这个假设只在关键词与诉讼标的完全重合时成立。
一旦关键词语义范围 > 一个诉讼标的（如"公司合并"包含吸收合并/新设合并/合并破产等多个诉讼标的），
LLM 被迫在异质的 case_steps 之间取交集，导致信息坍缩。

二、正确模型
================================================================================

关键词（大类根节点）
  │
  ├── 诉讼标的_A  →  子流程模板_A
  │     （来自 n 篇文章的 case_steps，按同一诉讼标的聚类后归纳）
  │
  ├── 诉讼标的_B  →  子流程模板_B
  │     （同上）
  │
  └── 诉讼标的_C  →  子流程模板_C
        （同上）

  辅助规则：
    - 不同诉讼标的之间的子模板高度相似时，可合并为一个子模板
    - 不与任何聚类相似的孤立 case_steps 保留为独立子模板

三、Stage3 改造方案
================================================================================

步骤 1：按 诉讼标的 聚类
  输入：Stage2 输出的所有 valid cases（每个 case 自带 诉讼标的 字段）
  方法：由 LLM 判断哪些 case 的诉讼标的属于同一范畴，归入同一聚类

步骤 2：聚类内归纳
  对每个聚类，将其内所有 case 的 case_steps 归纳为一个子流程模板
  模板结构保持 { nodes, edges } 格式不变

步骤 3：跨聚类合并
  如果多个子模板的结构高度相似（节点和边的重合度 > 阈值），合并为一个

步骤 4：输出树状模板结构
  顶层：关键词
  二级：诉讼标的（聚类标签）
  三级：子流程模板（{ nodes, edges }）

四、对现有流程的影响
================================================================================

  Stage2A：不变
  Stage2B：不变（已输出 诉讼标的 + 判决主文 + case_steps）
  Stage3： 从"一次 LLM 调用输出一个模板"改为"按聚类多次归纳，输出模板树"
  Stage3 输出格式：从单模板 JSON 改为模板树 JSON
